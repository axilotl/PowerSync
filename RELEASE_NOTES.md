## What's Changed

**Scheduled auto-update no longer silently skips days**
Multiple users reported that the overnight auto-update at 03:00 wasn't running despite the switch being on, with no log lines explaining why. Investigation on a live system found two failure modes the previous code couldn't surface or recover from. First, the integration fully reloads on any config-options change (battery spec auto-detection, hardware reserve sync, switch toggles, etc.), and each reload destroys the auto-update scheduler and creates a fresh in-memory closure with no record of whether today's run already happened. If a reload landed during the trigger minute (e.g. 03:00:30), the original tick fired against the now-dead listener and the new listener never matched the configured minute again that day. Second, the `homeassistant.update_entity` service refreshes the entity from HACS's already-cached data — it does not force HACS to re-fetch from GitHub. HACS has its own coordinator with its own check schedule, so when the scheduler fired at 03:00 the entity might still report no-update-available even when one had just been published.

The fix persists the "ran today" flag to disk so reloads can't reset it, widens the trigger window from "exactly the configured minute" to "any minute within four hours after the configured time, once per day" (so a reload at 03:00:30 still picks up today's run on the next minute-tick), retries the HACS refresh up to three times with a one-minute wait between attempts to give HACS room to actually re-fetch from GitHub, and adds diagnostic logging at every decision point so future debugging is possible without poking around in source. The Auto-Update PowerSync switch entity now also exposes `last_check_at` and `last_check_decision` attributes so anyone can verify their scheduler is alive and see why a recent run did or didn't fire.

If you previously turned auto-update on and saw nothing happen, this is the fix. After installing, check the switch's attributes — `last_check_at` should update every minute and `last_check_decision` will tell you what the scheduler is doing (`disabled`, `before_window`, `already_ran_today`, or `triggered`).
