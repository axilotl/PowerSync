## What's Changed

**Fix: BLE-only Tesla users are now visible to `discover_all_tesla_vehicles()` (#24, thanks @Artic0din!)**
When a user runs PowerSync with `ev_provider: tesla_ble` and no Fleet API / Teslemetry / Tessie integration, the automation planner's `discover_all_tesla_vehicles()` only scanned the HA device registry for `TESLA_INTEGRATIONS` identifiers and missed the ESPHome Tesla BLE bridges entirely (they register under the `esphome` domain with no VIN identifier). This silently broke two features for BLE-only users: `MultiVehicleEvaluator.evaluate_all_vehicles()` logged "No Tesla vehicles discovered" on every poll, and `_solar_surplus_switch_to_next_vehicle()` could never auto-advance to the second vehicle when the first finished charging. The planner now falls through to a Method 2 BLE discovery pass that reads `CONF_TESLA_BLE_ENTITY_PREFIX` and surfaces a vehicle for every configured prefix whose `binary_sensor.{prefix}_status` entity exists â€” the same canonical "BLE bridge is online" signal the mobile-app vehicle sync endpoint already uses. Gated on `ev_provider in (tesla_ble, both)` so Fleet APIâ€“only users see no behavior change.

Full credit to **@Artic0din** who diagnosed this on a dual-Model-Y PedroKTFC BLE setup and contributed the fix in PR #24.

**Fix: multi-BLE-prefix setups no longer silently ignore the second vehicle in default-VIN paths (issue #23, finding A)**
`is_ev_plugged_in()`, `get_ev_battery_level()`, `get_ev_location()`, and `_get_live_soc()` each had a `if "," in ble_prefix: ble_prefix = ble_prefix.split(",")[0].strip()` fallback that used only the first prefix in a comma-separated list when called with `vehicle_vin=None`. In a dual-car BLE setup (`ble_car1,ble_car2`) the second car was invisible to any "default vehicle" path: `is_ev_plugged_in(None)` returned False even with car2 plugged in, `get_ev_battery_level(None)` always returned car1's SOC, and `get_ev_location(None)` only checked car1's charger entity. All four call sites now iterate every configured prefix and return the first one with a valid signal (or True/home-location from any one, depending on function semantics).

**Cleanup: dead `tesla_ble`/`tesla_bluetooth` domain check removed (issue #23, finding B)**
A stale `if domain in TESLA_INTEGRATIONS or domain in ("tesla_ble", "tesla_bluetooth"):` check in `ev_charging_planner.py:4878` never actually matched anything â€” neither `tesla_ble` nor `tesla_bluetooth` is a real Home Assistant integration domain. ESPHome Tesla BLE bridges register under the `esphome` domain with empty `identifiers[]`, so the extra check was misleading dead code that suggested BLE support was wired up where it wasn't. Removed with a comment explaining why, so future readers aren't misled.

**Refactor: seven duplicated device-registry scan loops replaced with a shared helper (issue #23, finding C)**
The `ev_charging_planner.py` module had seven near-identical `for device â†’ for identifier â†’ is_tesla_vehicle = True â†’ break` loops scattered across `get_ev_location`, `discover_all_tesla_vehicles`, `is_ev_plugged_in`, `get_ev_battery_level`, `_get_live_soc`, `_ev_location_check`, and `is_plugged_in_check`. Each was ~15 lines of the same boilerplate. Extracted into a single `_iter_tesla_vehicle_devices(device_registry)` generator that yields `(device, vin)` tuples for every Tesla vehicle device, removing ~90 lines of duplication and ensuring any future BLE-style extension or new Tesla integration only needs to change one place. An incidental dead-code `device_name = device.name` assignment in the location scan was also dropped. No behavior changes.

**Fix: `force_charge` â‡„ `force_discharge` transitions now fully inherit the saved baseline (continuation of rpcai's PR #22 work)**
The cross-state clearing in 2.11.7 stopped `active` flags getting stuck but each transition still re-saved the intermediate force state as the "baseline", so when the second force mode auto-restored it reverted to the first force state rather than the true pre-force baseline. Both transition paths now inherit the `saved_tariff` / `saved_operation_mode` / `saved_backup_reserve` fields from the opposite state and skip re-saving, ensuring `force_charge â†’ force_discharge â†’ auto-restore` (and the reverse) returns the Powerwall to exactly the state it was in before *either* force command fired. Originally planned for 2.11.8 and merged into this release since it's the same subsystem.
