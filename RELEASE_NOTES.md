## What's Changed

**Flow Power Happy Hour: LP now actively grid-charges to 100% before HH when Maximum Profit Mode is on**
The optimizer's 48-hour rolling horizon was placing planned grid-charge slots at the globally cheapest PEA periods, which often left battery preparation for *today's* Happy Hour off the plan entirely — solar would top up to ~80% and stall, then HH would start with the battery short of full and significant export revenue left on the table. A new pre-window SoC floor now forces the LP to plan enough charging in the run-up to 17:30 so the battery reaches ~100% by HH start. The constraint doesn't dictate *when* to charge; the LP still picks the cheapest pre-HH slots itself (e.g. a deep PEA dip at 9 am beats a 4 pm slot), it just can't skip today entirely. Only active when Maximum Profit Mode is on and the provider is Flow Power; gracefully relaxes if the target isn't physically reachable in the time available.

**LP terminal valuation: corrected unit error so the optimizer properly values stored energy across day boundaries**
The terminal value coefficients on charge/discharge variables had a stray `/cap` factor that effectively shrunk terminal valuation by the battery capacity (~25× attenuation on a typical 25 kWh system). The result: the LP couldn't see the value of keeping reserve charge across the 48-hour horizon edge, so the battery would tend to drain toward horizon end and arrive at the next day's high-value export window underprepared. The math is now dimensionally correct ($/kWh × kW × h = $), which lets the LP properly plan cross-day storage — particularly noticeable for users on dynamic tariffs (Flow Power, Amber, Octopus) with deterministic high-export windows. Solar-equipped users see no behavior change for normal self-consumption because the LP's terminal price is already set from solar export rates (~5c) when solar is in horizon.

**Free electricity windows: LP economics now match the safety override**
GloBird FOUR4FREE and similar 0c-import periods previously relied on a hard override in the schedule builder to force max charging, because the LP's own `_free_charge_bonus` was attenuated by the same `/cap` unit error and the LP barely cared about free energy on its own. With the unit fix, the LP's economics now want to charge during 0c periods on their own (typically several c per kW per slot of incentive instead of fractions of a cent). The override remains in place as a safety net but is no longer load-bearing.
